# Neovim Keymap Inventory

Use this skill when you need to:
- document keymaps in this repo
- update `KEYMAPS.md`
- audit whether keymaps have descriptions
- distinguish explicit mappings from trigger-only plugin keys
- inspect dependency plugins used by this config when completeness matters
- avoid redoing dependency-plugin indexing work when the plugin version/revision has not changed
- include dependency/plugin behavior only when it can be justified from config review or runtime verification
- note when related plugins or features are explicitly disabled

## Source of truth

This repo keeps a manually maintained keymap document:

- `KEYMAPS.md`

It is **not** generated by a static parser.

`KEYMAPS.md` should use separate sections for:
- keymaps defined in this repo
- keymaps provided by dependency/downstream plugins that are active in this setup
- trigger-only or plugin-default entries that are relevant but should not be misrepresented as repo-defined mappings

## Recommended workflow

When completeness matters, use multiple discovery methods and reconcile them:

1. Code search in this repo
2. Code-aware review of the matching files
3. Inspect dependency plugins used by this config when relevant
4. Reuse prior dependency-plugin indexing/audit results if the plugin version/revision has not changed
5. Runtime validation in headless Neovim when needed
6. Update `KEYMAPS.md`

The LLM should use searches and file reads as discovery tools, but should not rely on a brittle static parser as the only source of truth.

## What to inspect

Look for:
- direct `vim.keymap.set(...)` mappings
- direct `vim.api.nvim_set_keymap(...)` mappings
- helper-wrapper mappings around those APIs
- plugin `keys = { ... }` specs declared in this repo
- contextual mappings created in callbacks, filetype hooks, `on_attach`, or plugin setup
- dependency plugin mappings or trigger definitions that affect active behavior
- related `enabled = false` settings that disable a plugin or a relevant feature

## Important limitations

This workflow is better than a naive static generator, but it still requires judgment.

That means:
- code search finds candidates, not final truth by itself
- helper wrappers and table composition must be interpreted semantically
- plugin `keys` entries may be true mappings or only lazy-load triggers
- dependency plugins may need separate inspection to understand inherited/default behavior relevant to this config
- if dependency-plugin inspection is repeated over time, keep track of plugin identity plus version/revision and reuse prior indexing/audit results when unchanged
- some mappings only exist in certain contexts, such as:
  - LSP-attached buffers
  - filetype-specific buffers
  - plugin windows or plugin-managed buffers
- when in doubt, validate by running Neovim headlessly and checking runtime keymaps

## Required workflow

Whenever editing Neovim config in this repo:

1. Ensure all user-facing mappings have a `desc`
2. Keep which-key groups aligned with active mappings
3. Audit relevant files with code search and code-aware review
4. If the mappings are dynamic/contextual, use runtime validation as needed
5. Update `KEYMAPS.md`
6. Review `KEYMAPS.md` for omissions, stale entries, and wrongly-classified trigger keys

## When auditing keymaps

Check both:
- direct mapping calls
- helper wrappers around `vim.keymap.set` / `vim.api.nvim_set_keymap`

Examples in this repo include:
- `lua/plugins/lsp.lua`
- `lua/plugins/neorg.lua`
- `lua/plugins/codecompanion.lua`

Also inspect plugin-spec `keys = { ... }` blocks and determine whether they are:
- explicit user-facing mappings
- lazy-load trigger keys
- plugin-default behaviors not explicitly defined in this repo

## Documentation expectations

When updating keymap docs:
- use separate sections for repo-defined mappings, dependency/downstream-plugin-defined mappings, and trigger-only/plugin-default entries
- clearly separate explicit mappings from trigger-only keys
- explicitly mention related plugins/features that are always disabled via `enabled = false`
- do not claim upstream plugin default mappings are repo-defined unless the repo actually defines them or runtime validation proves they are part of the active behavior you want documented
- prefer correctness and explanation over fake completeness from shallow parsing

## Dependency-plugin indexing guidance

When dependency-plugin inspection is needed:
- track plugin identity and version/revision/store hash for the dependency being inspected
- keep notes or cached indexing/audit results so unchanged dependencies do not need to be re-audited from scratch
- refresh dependency inspection when the plugin version/revision changes or when config changes make previously-irrelevant plugin behavior relevant
- treat dependency inspection as supporting evidence, not automatic proof that every discovered upstream keymap should be documented as active

## Files typically involved

- `lua/cfg/init.lua`
- `lua/plugins/*.lua`
- `lua/plugins/which-key.lua`
- `KEYMAPS.md`
- `AGENTS.md`

## Typical commands

Search for mapping definitions:

```bash
rg -n "vim\.keymap\.set|vim\.api\.nvim_set_keymap|keys = \{|keymap\(|nmap\(" lua --glob '!lua/plugins/graveyard/**'
```

Search for explicitly disabled features/plugins:

```bash
rg -n "\benabled\s*=\s*false\b" lua --glob '!lua/plugins/graveyard/**'
```

Optionally validate at runtime using headless Neovim for contextual mappings.

## Success criteria

A successful keymap-inventory update means:
- repo-defined user-facing mappings have descriptions
- `KEYMAPS.md` reflects the config after code-aware review
- disabled features/plugins relevant to keymaps are explicitly called out
- trigger-only keys are not misrepresented as explicit mappings
- which-key groups still match active keyspaces
